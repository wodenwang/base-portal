## Discovery 结论

### 目标用户

- Portal 运维者：需要把第三方系统按功能域、菜单、权限点接入门户。
- 企业内部用户：需要从门户无感进入已接入 Feishu IAM 的第三方系统。
- IAM / 平台维护者：需要确认权限点、权限组、审计和部署证据不泄露 secret。

### 核心问题

`v0.1.2` 已经证明 Portal 壳、菜单、tab、iframe 和部署链路可运行，但生产证据仍主要是健康检查和登录页 smoke。下一步要证明它能作为真实第三方应用接入层使用：真实 OAuth 登录、权限读取、菜单可见性、权限资源同步、审计和配置导入都能闭环。

### 约束

- 不把 `client_secret`、`developer_api_token`、access token、authorization code、cookie 或密码写入仓库、日志、截图或会话归档。
- `redirect_uri` 必须与 Feishu IAM 登记值完全一致。
- Portal 只做入口编排、登录态和权限过滤，不替第三方系统做二次鉴权。
- 管理后台先弱化；优先文件导入、seed、运维 API 和可验证脚本，不先做完整 CRUD 管理界面。
- Docker Compose、`install.sh`、`upgrade.sh`、版本 tag、线上读回必须继续遵守 `DEPLOY.md`。
- 前端若有改动，继续遵守 `DESIGN.md` 和既有 shadcn/Tailwind/Radix/lucide 基线。

### 成功标准

- 可以在受控环境完成真实 Feishu IAM OAuth 登录并回到 `https://base-portal.riversoft.com.cn`。
- 后端完成 code -> token -> userinfo -> `/api/v1/apps/base-portal/me/permissions` 链路，并用非敏感证据记录结果。
- 可以通过文件导入或运维 API 创建/更新至少一个第三方应用菜单配置，包含 domain、menu、url、openMode、resourceKey、confirmOnClose。
- IAM permission point/group 同步支持 dry-run 和 apply，输出只包含非敏感摘要、数量、key 和错误码。
- 登录用户的权限点能决定菜单可见性；无权限、部分权限、全权限都有可验证证据。
- 审计记录覆盖登录成功、菜单访问、IAM 资源同步、配置导入或变更。
- 发布和部署后读回 commit、tag、image、`.deploy/version`、`/health`、`/ready`、`/version`、首页、登录入口和真实 IAM 验收证据。

### 最小值得做切片

推荐切片：`一个第三方应用接入包 + 真实 IAM 验收闭环`。

内容包括：

- 定义一个无 secret 的配置文件格式，例如 `config/apps/*.json` 或 `config/portal-apps/*.json`。
- 导入一个第三方应用菜单，写入或更新 `PortalDomain` / `PortalMenu`。
- 校验 `resourceKey` 必须以 `base-portal.` 开头。
- 扩展 `/api/ops/sync-iam-resources`，支持 permission point/group dry-run 和 apply，记录审计。
- 用真实 Feishu IAM 登录读取用户信息和权限点，验证菜单过滤。
- 浏览器验证只覆盖 Portal 壳和嵌入入口，不强求第三方 iframe 页面本身移动端适配。

### 非目标

- 不做完整管理后台 CRUD。
- 不做通知、待办、最近访问、全局搜索、个性化首页等企业工作台能力。
- 不替第三方应用实现鉴权。
- 不引入第二套 UI 框架。
- 不在仓库中保存真实 env、token、cookie 或 secret。
- 不把 v0.1.2 #6-#9 作为 v0.2.0 功能范围。

### 主要风险

- 真实 OAuth 验收容易泄露 authorization code、access token 或 cookie，必须设计非敏感记录方式。
- 生产环境如果没有可授权测试账号，真实权限读取会被卡住，需要先定义 staging 或受控生产验证路径。
- IAM developer API 只允许维护本应用权限资源，不得越界修改应用配置、凭证、角色或管理员授权。
- 当前 session 是内存 Map，生产重启会丢会话；v0.2.0 是否要处理持久 session 需要决策。
- 当前 ops sync 只同步 permission points，若 v0.2.0 纳入 permission groups 和绑定，需要明确幂等行为和冲突处理。

## D1 - v0.2.0 范围选择

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 只做真实 IAM OAuth 验收 |  | 范围最小，最快补齐生产真实登录证据；对现有菜单和 UI 改动最少。 | 仍不能证明第三方应用接入运营化，只是证明登录链路能跑。 | `AuthService`、部署 env、验证文档、Playwright smoke |
| B. 真实 IAM 验收 + 一个第三方应用接入包 | 推荐 | 能证明 Portal 作为独立企业门户的真实价值；同时覆盖 OAuth、权限、菜单、IAM resource sync、审计和部署证据。 | 范围比 patch 大，需要严格切片，不能滑向完整管理后台。 | 后端 ops/import、Prisma/seed 或配置文件、前端少量状态、测试、部署 |
| C. 完整菜单/应用管理后台 |  | 后续运营体验最好，非技术人员可直接维护配置。 | 违背当前“管理后台先弱化”的边界，设计和权限范围会显著扩大。 | 新 UI、CRUD API、权限模型、审计、QA、设计评审 |

推荐：选择 B。它是 v0.2.0 的最小产品增强，不是单纯技术补证据，也不提前做大后台。

## D2 - 真实 IAM 验收环境

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 生产域名受控测试账号验收 | 推荐 | 最贴近真实用户路径，能直接证明 `base-portal.riversoft.com.cn` 可用。 | 需要测试账号、授权、权限点准备；记录证据时必须严控 secret 和 cookie。 | 生产 env、IAM 授权、Playwright/manual smoke、验证文档 |
| B. 本地或临时域名验收 |  | 调试快，失败成本低，不影响生产。 | `redirect_uri` 精确匹配会增加登记成本，不能完全替代生产验收。 | 本地 env、IAM 回调登记、临时部署或 tunnel |
| C. 只用 mock auth 回归 |  | 不需要真实账号和 secret，自动化最稳定。 | 不能满足 v0.2.0 的核心目标，仍缺真实 IAM 证据。 | 现有测试和 mock Playwright |

推荐：选择 A，但必须只记录非敏感证据，例如状态码、脱敏 user id、权限 key 数量、菜单可见性截图。

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 文件导入 + ops API | 推荐 | 符合“管理后台先弱化”；可 code review、可版本化、可 dry-run、可审计。 | 非技术人员不能直接操作，需要运维或开发协助。 | `config/`、ops controller/service、Prisma upsert、测试、文档 |

推荐：选择 A。它是从 seed 走向可运营化的中间层，且不会把 v0.2.0 拖成完整后台项目。
