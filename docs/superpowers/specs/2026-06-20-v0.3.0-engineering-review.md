# Base Portal v0.3.0 Engineering Review

日期：2026-06-20

## 结论

v0.3.0 可以进入 Step 6 `writing-plans`。工程方案采用小步收敛：不做数据库 schema migration，不改变部署拓扑，不让 Base Portal 代理第三方登录凭证；集中修正工作区状态模型、打开模式语义、iframe 刷新能力、SSO Demo 接入包和发布验证。

## 范围

包含：

- `#11` SSO Demo 菜单嵌入页无感访问路径。
- `#12` 移除沉浸模式。
- `#13` Tab 右键菜单和 Tab 操作菜单刷新当前 iframe。
- v0.3.0 版本号、release notes、验证文档、ship 和远端部署读回。

不包含：

- 完整第三方应用配置 UI。
- Base Portal 代理第三方 OAuth code、access token、cookie、client secret 或 developer API token。
- 删除数据库 `open_mode` 字段或做 schema migration。
- 修改 Feishu IAM 核心登录协议。

## D1：`new_tab` 数据语义

| 选项 | 结论 | Pros | Cons | 影响 |
|---|---|---|---|---|
| A. 严格按 `new_tab` 点击即浏览器新窗口 | 不采用 | 语义直观 | 打断 Portal 工作区，弱化 fallback 可恢复模型 | 前端行为变化大 |
| B. `new_tab` 作为 hint，菜单默认进入内部业务 Tab，新窗口由菜单/fallback 提供 | 采用 | 符合现有 `openMenuTab` 行为，保留统一 Tab 管理和审计 | 需要更新 `DESIGN.md` 和测试命名，避免文档误导 | 低风险 |
| C. 删除 `new_tab` 支持 | 不采用 | 最简单 | 破坏已有配置表达和审计上下文 | 不必要 |

实现要求：

- 前端菜单点击继续统一打开或激活内部业务 Tab。
- 业务 Tab 右键菜单、Tab 三点菜单和 fallback 面板提供“新窗口打开”。
- `recordMenuOpened` 使用实际业务 Tab openMode=`iframe` 记录；原始 menu openMode 可作为后续配置读回，不作为点击行为。

## D2：`immersive_iframe` 降级策略

| 选项 | 结论 | Pros | Cons | 影响 |
|---|---|---|---|---|
| A. 只隐藏按钮，保留状态函数和类型 | 不采用 | 改动少 | 容易留下不可达状态和测试漂移 | 技术债继续存在 |
| B. 前端运行态移除，历史数据降级为 iframe；接入包新数据规范化 | 采用 | 和 v0.3.0 设计一致，避免迁移风险 | 需要更新类型、测试、seed 或 import 逻辑 | 中低风险 |
| C. 数据库迁移删除历史值 | 不采用 | 数据最干净 | 影响生产数据，收益不足 | 高风险 |

实现要求：

- 删除或废弃前端 `enterImmersiveTab` / `exitImmersiveTab` 运行态和 UI 入口。
- `WorkspaceTab.openMode` 只允许 `iframe`。
- `NavigationMenu.openMode` 可继续接收后端原始值，但 `openMenuTab` 必须归一为 `iframe`。
- `OpsService` 接入包校验允许历史 `immersive_iframe`，但 import 时将其写入为 `iframe` 并在 summary warnings 中记录规范化提示。
- `apps/api/prisma/seed.ts` 中示例数据不再写入 `immersive_iframe`。

## D3：iframe refresh 状态模型

| 选项 | 结论 | Pros | Cons | 影响 |
|---|---|---|---|---|
| A. `window.location.reload()` | 不采用 | 简单 | 刷新整个 Portal，破坏工作区 | 明确禁止 |
| B. 每个业务 Tab 维护 refresh revision，iframe key 随 revision 变化 | 采用 | 可从 TabStrip 控制，保持 active tab/order | 需要扩展 `WorkspaceState` 和测试 | 低风险 |
| C. 只保留 fallback 面板刷新 | 不采用 | 改动小 | 不满足 #13 | 不完整 |

实现要求：

- 在 workspace model 中为每个业务 Tab 增加 `refreshKey` 或等价 revision。
- 新增 `refreshTabFrame(state, tabId)`，只递增目标业务 Tab 的 revision。
- `EmbedFrame` 使用 `tab.id + tab.refreshKey` 作为 iframe key。
- 右键菜单和 Tab 三点菜单均调用同一刷新动作。
- 无业务 Tab、首页 Tab、禁用状态时隐藏或禁用刷新动作。

## D4：SSO Demo 无感访问工程边界

| 选项 | 结论 | Pros | Cons | 影响 |
|---|---|---|---|---|
| A. Base Portal 注入 token/cookie/code 给 SSO Demo | 不采用 | 表面无感 | 严重违反安全边界 | 禁止 |
| B. Base Portal 提供标准 iframe + fallback；SSO Demo 自己优化嵌入模式 | 采用 | 职责清晰，可分项目迭代 | 需要 SSO Demo 配合才能完全无感 | 当前最稳 |
| C. Base Portal 只强制新窗口打开 | 不采用 | 规避 iframe 问题 | 放弃 Portal 内嵌体验 | 仅作 fallback |

实现要求：

- Base Portal 接入包新增或更新 SSO Demo 菜单，URL 指向 `https://feishu-iam-sso-demo.riversoft.com.cn/`，openMode 规范为 `iframe`。
- fallback 文案保持保守，不宣称识别具体原因。
- 生产验收记录只保留 HTTP status、页面可达、console/network 摘要和脱敏观察，不记录 token、cookie、authorization code 或真实用户身份。
- SSO Demo 配合需求单独沉淀到 `docs/external-requirements/2026-06-20-feishu-iam-sso-demo-embedded-sso.md`。

## 架构和数据流

```text
Feishu IAM login -> Base Portal session -> /api/navigation filters menus
  -> user opens SSO Demo menu
  -> Portal creates internal iframe tab
  -> iframe loads SSO Demo origin
  -> SSO Demo owns OAuth/session
  -> if iframe cannot become usable, Portal fallback/new-window recovery is available
```

Base Portal 不跨应用转发凭证。权限过滤仍使用 Base Portal 自己的权限点；SSO Demo 的应用权限由 SSO Demo 自己读取。

## 安全边界

- 不读取、不打印、不截图 `client_secret`、developer API token、OAuth code、access token、cookie、数据库密码或证书私钥。
- Base Portal 不新增跨应用 token exchange。
- SSO Demo 需求文档只能描述行为和验收，不包含真实 secret 或会话数据。

## Release / Deploy 风险

- `DEPLOY.md` 目前适用版本仍写 `v0.2.0`，v0.3.0 计划必须更新。
- 生产服务器为 amd64；构建镜像需继续使用 `docker buildx build --platform linux/amd64 ... --load` 或等价流程。
- 远端升级必须固定 `base-portal-release:v0.3.0`，并读回 `.deploy/version`、容器 image、`/version.commit`。
- 如果 SSO Demo 未完成配合改造，Base Portal v0.3.0 仍可发布“可恢复访问路径”，但“完全无额外交互”的验收应标记依赖 SSO Demo 迭代。
