## 工程决策

### D1 - 应用接入包形态

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. `config/portal-apps/*.json` + ops API 导入 | 推荐 | 配置可 review、可版本化、可 dry-run、可审计；符合不做管理 UI 的范围。 | 运维入口偏技术，需要文档和明确错误摘要。 | `config/`、ops service、Prisma upsert、测试、文档 |
| B. 继续只改 `seed.ts` |  | 改动最小，延续当前模型。 | 不是运营化入口，无法稳定处理 dry-run、差异和审计。 | `apps/api/prisma/seed.ts` |
| C. 数据库直改 / SQL 导入 |  | 可快速落地一次性数据。 | 风险高，难以审计和复用，容易绕开权限 key 校验。 | 运维脚本、DB |

决策：选择 A。

### D2 - IAM resource sync 覆盖范围

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 只同步 permission points |  | 延续现有 `/api/ops/sync-iam-resources`，风险最小。 | 不能证明权限组和绑定的开发者 API 闭环。 | `ops.controller.ts` |
| B. 同步 permission points、permission groups 和 group-point binding | 推荐 | 符合 `AGENTS.md` 接口清单，能证明接入包完整进入 IAM 权限模型。 | 实现和测试更复杂，需要清晰幂等和错误摘要。 | ops service、配置 schema、测试、文档 |

决策：选择 B。v0.2.0 至少支持一个接入包中的 group 和 group-point binding dry-run/apply。

### D3 - 数据库 schema

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 不新增表，使用现有 `PortalDomain`、`PortalMenu`、`PortalAuditEvent` | 推荐 | 当前模型足以表达 domain/menu/resourceKey/url/openMode；避免不必要 migration。 | 配置包版本只能记录在审计 detail 中，不能查询完整历史实体。 | ops import、audit detail、测试 |
| B. 新增 `PortalAppPackage` / `PortalSyncRun` 表 |  | 长期审计和回滚更强。 | 超出 v0.2.0 最小切片，会扩大迁移和部署风险。 | Prisma migration、服务、测试 |

决策：选择 A。本版不加 schema，除非实现时发现现有模型无法表达必要字段。

### D4 - session 持久化

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 保持内存 session，记录生产重启需重新登录 | 推荐 | 不扩大 v0.2.0 范围，真实 IAM 验收仍可完成。 | 容器重启后用户会话失效；不能多实例横向扩展。 | 文档、验证说明 |
| B. 引入数据库 session |  | 更接近生产长期形态。 | 增加 schema、清理策略、安全和测试复杂度。 | Prisma migration、session service、测试 |

决策：选择 A。v0.2.0 验收目标不是会话持久化；但发布说明必须写清限制。

### D5 - 真实 IAM 验收证据

| 选项 | 推荐 | Pros | Cons | 影响范围 |
|---|---|---|---|---|
| A. 生产受控账号 + 脱敏证据 | 推荐 | 最接近真实用户路径，能证明生产域名可用。 | 需要测试账号、权限准备和严格脱敏。 | 验证文档、Playwright/manual smoke |
| B. 仅本地/临时域名 |  | 调试快。 | 不能替代生产验收，redirect_uri 还要额外登记。 | 临时部署、IAM 配置 |
| C. 仅 mock auth |  | 自动化稳定。 | 不满足 v0.2.0 核心目标。 | 现有测试 |

决策：选择 A。证据只记录阶段、状态码、脱敏 user id、权限 key 数量/非敏感 key、菜单可见性和截图，不记录 code/token/cookie/secret。

## API 合同

### `POST /api/ops/import-app-package`

认证：必须有 Portal session，且 `session.user.id` 在 `PORTAL_ADMIN_USER_IDS` 中。

请求体：

```json
{
  "dryRun": true,
  "package": {
    "packageKey": "base-portal.demo",
    "version": "2026-06-19",
    "domain": {},
    "menus": [],
    "permissionGroups": []
  }
}
```

响应只返回非敏感摘要：

```json
{
  "ok": true,
  "dryRun": true,
  "summary": {
    "domains": { "create": 1, "update": 0 },
    "menus": { "create": 3, "update": 0, "disable": 0 },
    "permissionPoints": 2,
    "permissionGroups": 1,
    "bindings": 2,
    "warnings": []
  }
}
```

### `POST /api/ops/sync-iam-resources`

请求体：

```json
{
  "dryRun": true,
  "packageKey": "base-portal.demo"
}
```

响应只返回 key、状态、数量和错误码；不返回 token 或 IAM 原始敏感响应。

## 测试策略

后端：

- 配置包校验：namespace、level、leaf/url/resourceKey、openMode、重复 key。
- dry-run 不写 DB、不调用 IAM。
- apply upsert domain/menu，重复执行幂等。

- NavigationService 覆盖无权限、部分权限、全权限。

## 发布风险

- 远端 Docker pull 可能不稳定；沿用 `--pull never` 和远端已加载 image 的升级路径。
- 真实 IAM 登录如果权限未授权，会显示空权限状态；这是可接受验收分支，不应误判为系统故障。
- 第三方 URL 可能因 CSP / X-Frame-Options 不允许 iframe；Portal 只提供 fallback 和新窗口打开，不承诺强行嵌入。
- 当前 session 重启丢失；发布说明必须写清 v0.2.0 未解决持久 session。
