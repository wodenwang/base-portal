# Base Portal v0.3.0 Product Design Review

日期：2026-06-20

Step 4 通过。`design/2026-06-20-v0.3.0-design-decision.md`、`DESIGN.md`、`design/base-portal-visual-target.md` 和 `design/base-portal-final-design-option-2.md` 已足以支撑 v0.3.0 工程评审。

本版本不需要新增完整 Product Design 视觉稿，也不需要 Pencil 原型。v0.3.0 的主要变化是交互与状态治理，不改变 `Light Command Workspace` 的视觉架构。

## 复核项

### 1. SSO Demo 无感访问责任边界

通过。

设计制品已经明确：

- Base Portal 维护菜单入口、Portal 登录态、权限过滤、Tab 工作区、iframe 承载和 fallback。
- SSO Demo 自己完成 Feishu IAM OAuth、session、用户信息读取和权限读取。
- Base Portal 不传递、不代理、不暴露第三方应用的 OAuth code、access token、cookie、client secret 或 developer API token。

后续工程评审需要确认 SSO Demo 在 iframe 和新窗口两种打开方式下的实际 cookie / OAuth 行为，并明确生产验收方式。

### 2. 移除沉浸模式

通过。

`DESIGN.md` 已不再把沉浸模式列为当前核心职责、组件模式或状态覆盖。当前设计治理要求：

- 用户可见 UI 不再出现“沉浸模式”入口、文案、tooltip、右键菜单项或三点菜单项。
- 业务页面只保留标准 iframe 和应用内最大化两种工作区形态。
- 历史 `immersive_iframe` 数据运行时兼容降级为标准 iframe。

### 3. 历史 `openMode=immersive_iframe` 兼容降级

通过。

设计层已明确不做 schema migration，允许数据模型继续保存历史字符串；前端和产品能力不再呈现沉浸模式。后续工程评审需要决定：

- 是否在接入包校验中警告或规范化 `immersive_iframe`。
- 是否在前端类型中移除业务运行态对 `immersive_iframe` 的依赖。
- 是否需要保留 API 原始 `openMode` 字段用于审计或配置读回。

### 4. 刷新当前 iframe

通过。

设计层已覆盖：

- 桌面业务 Tab 右键菜单提供刷新当前 iframe。
- 平板和手机 Tab 操作三点菜单提供同等刷新能力。
- 刷新只作用于当前业务 Tab 对应 iframe，不刷新整个 Portal，不改变 Tab 顺序，不关闭页面，不破坏当前 active Tab。
- 首页固定 Tab、无业务 Tab 或非 iframe 页面时，刷新动作隐藏或禁用。
- fallback 面板中的刷新动作继续保留，并与 Tab 菜单刷新共用底层行为。

### 5. 是否需要新增完整视觉稿

不需要。

理由：

- v0.3.0 不改变 AppShell、TopBar、SidebarTree、TabStrip、EmbedFrame 的整体布局。
- v0.3.0 不新增页面、面板、复杂表单或完整配置 UI。
- 当前视觉变化只是在既有 Radix context menu / dropdown menu 中增删动作，以及删除沉浸入口。
- `Light Command Workspace` 已覆盖工作区、Tab、iframe、fallback、最大化和响应式基线。

## Step 5 工程评审输入

进入 `plan-eng-review` 前，必须带入以下工程问题：

1. `new_tab` 数据语义：`DESIGN.md` 仍保留 `new_tab` 不占 Portal 内部 Tab 的规则，而 v0.3.0 设计倾向是菜单默认先进入内部业务 Tab，失败时提供新窗口恢复。工程评审必须决定是否同步调整 `DESIGN.md` 文案、数据语义或运行策略。
2. `immersive_iframe` 降级策略：前端状态、类型、测试、seed 数据、接入包校验和 docs 是否需要统一收口。
3. iframe refresh 状态模型：是否以 `frameVersion` / tab revision / iframe key 等方式表达，确保刷新当前 iframe 不影响 Portal shell。
4. SSO Demo 无感访问：需要检查 SSO Demo 自身 OAuth/session/cookie 策略，确认 iframe 和新窗口 fallback 的真实行为；Base Portal 不应处理第三方 secret 或 token。
5. 验证矩阵：单测覆盖 workspace 状态、组件测试覆盖菜单动作，Playwright 覆盖桌面右键菜单、移动三点菜单、iframe refresh、无沉浸入口和生产 SSO Demo 路径。
