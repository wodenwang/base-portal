# Base Portal v0.3.0 设计制品判断

日期：2026-06-20

## 结论

v0.3.0 不新增完整 Product Design 视觉稿，也不需要 Pencil 原型。当前 `Light Command Workspace` 仍是有效视觉目标；本版本只调整 Portal 壳的交互规则和恢复动作，不改变整体信息架构、布局密度、色彩系统或组件风格。

本文件作为 v0.3.0 的轻量设计制品，优先级高于早期设计资产中关于“沉浸模式”的描述。后续实现必须同步遵守 `DESIGN.md` 的 v0.3.0 规则。

## 版本范围

- `#11`：SSO Demo 菜单嵌入页必须实现无感访问。
- `#12`：移除沉浸模式。
- `#13`：Tab 右键菜单和 Tab 操作菜单增加刷新当前 iframe。

## 责任边界

Base Portal 的职责：

- 维护菜单入口、Portal 登录态、权限过滤、Tab 工作区、iframe 承载和 fallback。
- 在第三方页面不能稳定嵌入或不能静默完成登录时，提供清晰、可恢复的打开方式。
- 不传递、不代理、不暴露第三方应用的 OAuth code、access token、cookie、client secret 或 developer API token。

SSO Demo 的职责：

- 自己完成 Feishu IAM OAuth、session、用户信息读取和权限读取。
- 在被 Portal iframe 或新窗口打开时，提供可恢复的登录路径。
- 不依赖 Base Portal 向它注入任何敏感凭证。

Feishu IAM 的职责：

- 提供统一 OAuth 授权、用户信息和应用权限读取能力。
- 通过浏览器已有 IAM 登录态支持第三方应用尽量减少重复登录交互。

## 交互决策

### SSO Demo 入口

- Portal 菜单仍可打开 SSO Demo 业务 Tab。
- 首选在标准 iframe 中承载；如果 iframe 内部无法完成可用访问，fallback 必须提供新窗口打开和复制链接。
- fallback 文案只能表达“页面可能无法加载或需要新窗口访问”，不得声称准确识别 CSP、cookie、OAuth state 或跨域原因。
- 真实验收必须覆盖生产域名 `https://base-portal.riversoft.com.cn` 到 `https://feishu-iam-sso-demo.riversoft.com.cn/` 的用户路径。

### 移除沉浸模式

- v0.3.0 移除所有用户可见的“沉浸模式”入口、文案、tooltip、右键菜单项和三点菜单项。
- 业务页面只保留标准 iframe 和应用内最大化两种工作区形态。
- 历史菜单数据中的 `openMode=immersive_iframe` 不作为产品能力继续呈现；运行时应兼容降级为标准 iframe。
- 不为本次改动做高风险 schema migration；数据模型可继续保存历史字符串，但前端和设计不再把它当成当前能力。

### 刷新当前 iframe

- 桌面：业务 Tab 右键菜单增加“刷新”动作。
- 平板和手机：Tab 操作三点菜单增加“刷新”动作。
- 刷新必须只作用于当前业务 Tab 对应的 iframe，不刷新整个 Portal，不改变 Tab 顺序，不关闭页面，不破坏当前 active Tab。
- 首页固定 Tab、无业务 Tab 或非 iframe 页面时，刷新动作应隐藏或禁用。
- fallback 面板中的刷新动作继续保留，并与 Tab 菜单刷新共用同一底层行为。

## 响应式要求

- 桌面 1440px：右键菜单可发现刷新、新窗口、最大化、关闭和移动动作；不再出现沉浸模式。
- 平板约 834px：三点菜单可完成刷新、新窗口、最大化和关闭动作；菜单项不换行。
- 手机约 390px：Portal 壳不破版；三点菜单提供当前 iframe 刷新和新窗口 fallback；不承诺第三方 iframe 内部页面适配。

## 状态覆盖

v0.3.0 的设计和验证必须覆盖：

- 已登录后打开 SSO Demo 菜单。
- SSO Demo iframe 正常可访问。
- SSO Demo iframe 需要 fallback 或新窗口打开。
- 当前 iframe 从右键菜单刷新。
- 当前 iframe 从三点菜单刷新。
- 最大化进入和退出。
- 历史 `immersive_iframe` 数据降级为标准 iframe。
- 桌面、平板、手机视口下无沉浸模式残留入口。

## 不做事项

- 不新增完整管理后台。
- 不新增第三方应用配置 UI。
- 不引入第二套 UI 框架。
- 不把 SSO Demo 的视觉系统并入 Portal。
- 不让 Base Portal 持有或转发第三方系统的敏感凭证。
- 不做 schema migration 来删除 `openMode` 历史取值。

## 后续进入工程评审前的问题

- 是否需要 Base Portal 在接入包校验中警告或规范化 `immersive_iframe`。
- SSO Demo 当前 `SameSite=Lax` session cookie 在 iframe 场景下是否会限制静默访问；若限制，应优先由 SSO Demo 自己调整或提供新窗口恢复路径。
- Tab 刷新动作应如何在 React 状态模型中表达，避免把局部 iframe refresh 变成整个应用 reload。
- 生产验收如何脱敏记录 OAuth、session、cookie 和 network 证据。
